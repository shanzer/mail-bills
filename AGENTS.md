# Repository Guidelines

## Project Structure & Module Organization

This repository is a TypeScript/Node.js rewrite of the Mail Bills physical-mail pipeline.

- `src/` contains application code.
  - `api.ts` exposes the Fastify REST API.
  - `cli.ts` wires command-line entry points.
  - Pipeline modules include `intake.ts`, `importer.ts`, `ocr.ts`, `classifier.ts`, `dedupe.ts`, `processor.ts`, `review.ts`, and `quarantine.ts`.
  - External integrations live in `notion.ts`, `reminders.ts`, and `messaging.ts`.
  - Shared types and helpers live in `types.ts`, `config.ts`, `paths.ts`, and `ledger.ts`.
- `tests/` contains Vitest unit and integration-style tests, named `*.test.ts`.
- `dist/` is generated by TypeScript build output and should not be edited by hand.
- `README.md` documents runtime commands and API routes.

## Build, Test, and Development Commands

- `npm install` installs dependencies.
- `npm run check` runs TypeScript type checking without emitting files.
- `npm test` runs the Vitest suite.
- `npm run build` compiles TypeScript into `dist/`.
- `npm run start -- --host 127.0.0.1 --port 8788` starts the compiled REST API.
- `npm run mail-bills -- pairing --config ../mail-bills/config.yaml --base-url http://yoyodyne:8765 --qr` generates iPhone pairing output.

Run `npm run build` before using `npm run start` or `npm run mail-bills`, because those commands execute `dist/src/cli.js`.

## Coding Style & Naming Conventions

Use strict TypeScript with ES modules. Prefer explicit interfaces for API payloads, adapter results, and ledger records. Use two-space indentation, `camelCase` for TypeScript symbols, and `snake_case` only where preserving the existing SQLite schema or external payload contract.

Keep side effects injectable where practical, especially for Notion, reminders, messaging, and processor orchestration. Avoid shelling out from UI or API code unless the module is specifically an adapter.

## Testing Guidelines

Vitest is the test framework. Add focused tests in `tests/*.test.ts` for new behavior, especially around upload safety, ledger compatibility, pipeline dry-runs, and reversible delete/quarantine flows.

Tests should use temporary directories outside the repo via helpers in `tests/helpers.ts`. Do not mutate `../mail-bills/data/ledger.sqlite` or live intake folders from tests.

## Commit & Pull Request Guidelines

No repository Git history is available here, so use clear, imperative commit messages such as `Add pairing QR CLI` or `Harden status endpoint path checks`.

Pull requests should include:

- A short description of behavior changed.
- Commands run, usually `npm run check`, `npm test`, and `npm run build`.
- Any config or security implications, especially token, upload, Notion, reminder, or file-retention changes.

## Security & Configuration Tips

The iPhone intake endpoint uses a shared bearer token from `intake_upload.token`. Never log or expose tokens beyond pairing output. Use a phone-reachable base URL such as `http://yoyodyne:8765`, a LAN IP, or Tailscale address; `127.0.0.1` only works from the Mac itself.
